# Overture V1.1 — State of the Build & Roadmap

*Audited 2026-06-10. Perspectives: product, engineering, UX. This document is the durable source of truth for what exists, what's broken, and what rounds out V1.1.*

---

## 1. What we've built (and it's a lot)

### Actor side — strong, near-complete
- **Discover** — radius/sort/type filters, search, promoted listings, "For You" vs "Further Out" split, profile-driven defaults. Complete.
- **Audition detail** — phase-aware state machine (browsing → signed up → callback pending/confirmed → cast offered/accepted). Best page in the app. Now public for anonymous visitors (browsing phase).
- **Signup modal** — slot picker with live availability, role interest, conflict date picker, acknowledgments.
- **My Shows** — per-show journey timelines, callback accept/decline, past credits. Complete.
- **Notifications** — unread/read split, mark-as-read, type-based icons. Complete.
- **Cast offers** — celebratory accept flow with agreements, confetti, contact block. Complete.
- **Profile** — single scrolling page; bucket list, vitals, three-tier privacy (public / private / hidden), minor badge + guardian card, endorsements, verified vs manual credits, training, awards, measurements. Editing via four focused modals (Profile / Credits / Training / Awards) with item-level saves.
- **Onboarding** — fork (actor / theatre-maker), full actor wizard incl. minor/guardian branch, theatre-maker wizard creating an org, simulated Google OAuth that pre-fills identity, login redirect for incomplete onboarding.

### Production side — workflow-complete for a solo director
- **Show creation wizard** (4 steps: details, roles, schedule, team).
- **Setup hub** — full CRUD on details/roles/slots/team, phase-transition guardrails, live stats.
- **Auditions** — slot-grouped schedule, batch status ops, search/filters, actor slide panel with team notes.
- **Callbacks** — role-based pools, uncalled-actors safety net, prep notes, stats.
- **Casting board** — primary/alternate/understudy, candidate pools prioritizing accepted callbacks, unfilled-role warnings, send offers → publish gating.
- **Cast list** — celebratory presentation, share/print, unpublish.
- **Phase-gated sub-nav** that unlocks tabs as the show progresses.

### Platform
- **Public layer (new)** — `(public)` route group: `/browse`, `/auditions/[id]`, `/theatres/[orgId]`, all anonymous-viewable with auth-aware Nav (Sign in / Sign up when logged out).
- **Design system** — consistently applied (both audits found zero violations). CLAUDE.md is the enforced source of truth.
- **Mock API layer** — clean seam for Supabase migration; schema exists in migrations.

## 2. The honest framing

The app is a **superb single-player demo of a multiplayer product**. Every screen works, but the moments where one human's action should reach another human are mocked:

- "Send Notifications" (callbacks) → no-op
- "Send Offers" (casting) → no-op
- "Publish Cast List" → status change, nobody told
- "Add Team Member" → fake local user, no invite, invitee can never log in
- All data resets on refresh (mock layer — by design until Supabase)

A real theatre with a real team and real actors would discover within an hour that nothing actually reaches anyone. That's the V1.1 gap in one sentence: **make the connective tissue real.**

**Definition of V1.1 done:** a real theatre with a 3-person team can onboard, create a show, run auditions with real actors who signed up via a shared public link, send callbacks and offers that actors actually receive, and publish a cast list — with all data surviving a refresh, minors protected by guardian consent, and private data enforced by RLS.
